## One container, two notions of “where we are”

`LocomotiveScroll` is constructed on the element marked `data-scroll-container`. With `smooth: true` and a low `lerp` (here `0.13`), the library interpolates toward a target scroll position instead of snapping the DOM to native scroll every frame. Options such as `smartphone: { smooth: true }` and `tablet: { smooth: true }` extend that behavior to coarse pointers so the demo does not feel “broken on mobile” while still being a **scroll hijack** you should treat carefully for accessibility.

Inside that container, layout is a row of `.column-wrap` blocks, each holding a `.column` of `figure.column__item` cells. The middle column carries `data-scroll-section`, which Locomotive uses to segment the page for its own scroll-linked features. The outer columns are ordinary structure; they become special only in JavaScript.

## Selecting “odd” columns

```js
const oddColumns = [...document.querySelectorAll('.column')].filter(
  (_, index) => index != 1,
)
```

`querySelectorAll('.column')` returns **all three** vertical stacks in document order. The filter drops `index === 1`, i.e. the **center** column. What remains is the left and right rails—the elements that will receive a manual `transform`, while the middle column’s motion comes almost entirely from Locomotive’s own translation of content inside the smooth root.

Naming is a small cognitive tax: “odd columns” here really means **outer** columns (indices `0` and `2`), not odd-numbered DOM indices in the mathematical sense.

## The entire parallax contract in one line

```js
scroll.on('scroll', (obj) => {
  oddColumns.forEach(
    (column) => (column.style.transform = `translateY(${obj.scroll.y}px)`),
  )
})
```

On every scroll tick, `obj.scroll.y` is the library’s **virtual** vertical scroll value. Writing it into `translateY` on the outer columns **stacks** a transform on top of whatever Locomotive already applies to ancestors. The center column skips this assignment, so it does not get the same stacked offset. Visually, that mismatch reads as **differential speed** between the middle gallery and the wings—exactly the “alternate column” affordance without hand-rolling a second scroll timeline.

## Markup roles (minimal reading)

The root `.columns` element doubles as `data-scroll-container`: Locomotive owns smooth scrolling for everything inside. Outer wrappers use a modifier like `column-wrap--height` in the full CSS to establish **tall** side columns versus the shorter middle track, which increases how much relative motion you see before the section leaves the viewport. Image cells use nested wrappers (`column__item-imgwrap`, `column__item-img`) so aspect ratio, overflow clipping, and optional parallax on the inner image stay independent of the column-level `translateY`.

`data-pos` attributes order items for the **detail** interaction in the complete demo; they are not required for the two-line parallax hook.

## ESM.sh and production hygiene

```js
import LocomotiveScroll from 'https://esm.sh/locomotive-scroll@4.1.3'
```

That import is ideal for CodePen-style sandboxes. In an app bundled with Vite or webpack, prefer `locomotive-scroll` from npm so you control version ranges, tree-shaking, and Content Security Policy. Either way, wire **resize** and **DOM mutations** to Locomotive’s `update()` method when column heights change, and **destroy** the instance when the route unmounts so listeners and `requestAnimationFrame` work do not leak.

## Accessibility and motion budget

Smooth scroll libraries fight native scroll anchoring, keyboard step sizes, and `prefers-reduced-motion`. A practical split:

- When `prefers-reduced-motion: reduce` is set, skip `smooth: true` and skip the `translateY` coupling so the grid degrades to a normal stacked layout.
- Keep focusable controls out of aggressively transformed subtrees if you see focus rings drifting from visible hit targets in Safari.

## What to copy first

- **Outer columns selected explicitly** (here by index) so the center column remains the visual anchor.
- **A single scroll emitter** (`scroll.on('scroll', ...)`) driving cheap `transform` updates—no second physics loop.
- **Tall side tracks + shorter middle** in CSS so the parallax has room to read before the section scrolls away.

What to verify before shipping: scroll restoration on back navigation, interaction with sticky headers, and whether your analytics or third-party widgets assume native `window` scroll events (Locomotive often means those need bridging).
